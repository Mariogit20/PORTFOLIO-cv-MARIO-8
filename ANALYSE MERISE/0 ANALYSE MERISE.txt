PHASE 3 : Le Modèle Logique de Données (MLD)
C'est la traduction du MCD vers une structure de base de données relationnelle. C'est ici qu'apparaissent les Clés Primaires (PK) et les Clés Étrangères (FK).

Règles de passage de MERISE (MCD -> MLD) :

Règle du (1,1) ou (0,1) : L'entité côté (1,1) reçoit la clé primaire de l'autre entité en tant que clé étrangère.

Règle du (0,N) - (0,N) : L'association se transforme en une nouvelle table (table de jonction) qui prend pour clé primaire composée les clés des deux entités reliées.

Le Schéma Logique (correspondant à vos modèles Django) :

ROLE (<u>id_role</u>, nom_role)

UTILISATEUR (<u>id_user</u>, nom, email, password, #id_role)

Commentaire : L'application de la règle 1 place l'id_role ici (ce qui correspond à votre models.ForeignKey(Role)).

MENU (<u>id_menu</u>, nom, code_menu)

ROLE_MENU_ACCES (<u>#id_role, #id_menu</u>, est_visible)

Commentaire : Application de la règle 2. L'association AUTORISER devient cette table. Dans Django, c'est exactement ce que vous avez fait avec l'option unique_together = ('role', 'menu').

HISTORIQUE (<u>id_hist</u>, action, date_action, app_label..., #id_user)

Commentaire : #id_user peut être nul (visiteur anonyme).

REALISATION (<u>id_real</u>, nom, description)

CLIC_DEMO (<u>id_clic</u>, clicked_at, ip, session_key, #id_realisation)

PHASE 4 : Le Modèle Physique de Données (MPD)
Le MPD est l'implémentation finale dans un SGBD spécifique (ici, SQLite via Django ORM).

C'est ici qu'on définit l'optimisation physique :

Les Index : Dans votre MERISE physique, vous avez ajouté des index (ex: db_index=True sur action, app_label et la création de l'index app_acceuil_realisa_8b0be6_idx pour les clics). C'est crucial dans le MPD pour accélérer les recherches (les clauses WHERE).

Les Types de données : Vos identifiants id deviennent des BigAutoField (entiers longs auto-incrémentés). Les textes deviennent des VARCHAR (pour les CharField avec max_length) ou des TEXT (pour les TextField).

PHASE 5 : Le Modèle Conceptuel des Traitements (MCT) - Le petit plus
MERISE ne fait pas que des données, elle analyse aussi les processus. Prenons l'exemple de votre Middleware d'Historique (le suivi des lectures de pages) :

Événement déclencheur : Le serveur reçoit une requête HTTP.

Synchronisation : La requête doit être de type GET ET l'URL ne doit pas commencer par /static/ ou /media/ ET le Header doit demander du text/html.

Opération (Le Traitement) : Extraction de l'IP, du User-Agent, vérification de la Session et de l'utilisateur connecté (_local.user).

Résultat A (Condition remplie) : Création d'un enregistrement dans la table HISTORIQUE avec l'action "READ". L'événement de sortie est : "Page tracée".

Résultat B (Condition non remplie) : Fin du processus sans enregistrement (ex: c'était juste le chargement d'une image). L'événement de sortie est : "Requête ignorée".

Aucune redondance de données (grâce au passage aux identifiants).

LÉGENDE ET RÈGLES DE LECTURE MERISE :
Le nom de la table est en MAJUSCULES.

Les Clés Primaires (CP) sont <u>soulignées</u> (ce sont les identifiants uniques).

Les Clés Étrangères (CE) sont précédées d'un hashtag # (ce sont les liens vers les autres tables).

Les types de données globaux sont précisés pour faire le lien avec l'intégration technique.

Dans votre implémentation Django, le framework a généré un identifiant technique id_role_menu (BigAutoField) tout en appliquant une contrainte UNIQUE sur le couple Rôle/Menu. Après avoir établi le Modèle Conceptuel des Traitements (MCT), la prochaine étape logique et chronologique dans la méthode MERISE est le Modèle Organisationnel des Traitements (MOT).

Si le MCT répondait à la question "QUOI ?" (Quelles sont les règles de gestion et les opérations à effectuer, abstraction faite de la technique), le MOT descend d'un niveau pour répondre aux questions "QUI ?", "OÙ ?", "QUAND ?" et "COMMENT ?".

Voici comment cette étape fonctionne et s'applique à votre architecture.

1. Le rôle du Modèle Organisationnel des Traitements (MOT)
Le MOT reprend exactement les mêmes processus que le MCT, mais il y ajoute les contraintes d'organisation de l'entreprise (ou ici, de votre architecture web). On va répartir les tâches définies dans le MCT entre les différents acteurs (humains ou machines) et définir la nature de l'interaction.

Dans un MOT classique, on ajoute ces colonnes/informations à notre processus :

L'Acteur / Le Poste de travail (QUI / OÙ) : Est-ce le Visiteur sur son navigateur ? L'Administrateur sur son tableau de bord ? Le Serveur Django en arrière-plan ?

Le Temps (QUAND) : L'action est-elle immédiate (temps réel), différée, ou exécutée à une date précise (batch/tâche planifiée) ?

La Nature du traitement (COMMENT) :

Manuel (M) : Une action humaine sans machine (ex: réfléchir à un mot de passe).

Interactif (I) : Un dialogue entre l'humain et la machine (ex: remplir et valider un formulaire HTML).

Automatique (A) : Une tâche exécutée par la machine seule (ex: hacher le mot de passe et faire l'INSERT en base de données).

DICTIONNAIRE DES RÈGLES DE GESTION (RG)
1. DOMAINE : UTILISATEURS ET SÉCURITÉ (app_user & app_contact)
RG_SEC_01 : Unicité de l'identité. Un utilisateur est identifié de manière unique dans tout le système par son adresse email. Il est impossible de créer deux comptes avec la même adresse.

RG_SEC_02 : Normalisation des emails. Toute adresse email saisie lors de la création d'un compte doit être formatée (minuscules, suppression des espaces) avant son enregistrement en base.

RG_SEC_03 : Sécurité des mots de passe. Le mot de passe d'un utilisateur ne doit jamais être stocké en clair. Il doit obligatoirement subir un hachage cryptographique (algorithme PBKDF2) avant insertion.

RG_SEC_04 : Obligation de Rôle. Tout utilisateur enregistré dans le système doit obligatoirement être rattaché à un, et un seul, Rôle défini (Administrateur, Utilisateur, etc.).

2. DOMAINE : HABILITATIONS ET MATRICE RBAC (app_user)
RG_HAB_01 : Contrôle d'accès par rôle. L'accès aux différents modules et menus de l'administration est strictement conditionné par le rôle de l'utilisateur connecté.

RG_HAB_02 : Génération automatique des droits (Signaux). Lors de la création d'un nouveau Rôle ou d'un nouveau Menu, le système doit générer automatiquement les lignes de croisement dans la matrice des permissions, avec une valeur d'accès refusé (False) par défaut.

RG_HAB_03 : Règle d'inaliénabilité de l'Administrateur. C'est une sécurité absolue codée en dur : le rôle "Administrateur" possède de manière permanente et inaliénable le droit d'accès au menu "Gestion des Utilisateurs". Cette permission force la valeur True lors de toute mise à jour pour éviter un auto-blocage du système.

3. DOMAINE : TRAÇABILITÉ ET AUDIT (app_historique)
RG_AUD_01 : Suivi de navigation. Toute requête de lecture (GET) aboutissant à une page HTML doit générer une trace dans l'historique (exclusion faite des appels aux fichiers statiques et médias).

RG_AUD_02 : Traçabilité hybride. Une trace d'historique enregistre l'adresse IP et le User-Agent. Elle est rattachée à un utilisateur si celui-ci est connecté, ou conservée de manière anonyme dans le cas d'un visiteur public.

RG_AUD_03 : Durée de conservation (RGPD). Le système paramètre la durée de conservation légale des logs d'historique (par défaut fixée à 2 ans). Ce paramètre est unique (Singleton).

RG_AUD_04 : Purge algorithmique. La procédure de purge supprime de manière irréversible toutes les traces d'historique dont l'année d'enregistrement est strictement inférieure à l'année courante soustraite de la durée de conservation.

4. DOMAINE : PORTFOLIO ET ANALYTIQUE WEB (app_acceuil)
RG_POR_01 : Visibilité par défaut. Tout nouvel élément ajouté au portfolio (Réalisation, Compétence, Réseau Social) est considéré comme visible (est_visible = True) pour le public dès sa création.

RG_POR_02 : Métriques de Démo. Chaque clic d'un visiteur sur le lien "Démo Live" d'une réalisation doit incrémenter le compteur analytique de ce projet de +1.

RG_POR_03 : Audit de Démo. Outre l'incrémentation du compteur, chaque clic de démo génère un événement d'audit distinct stockant l'horodatage précis, l'IP et la clé de session du visiteur.

RG_POR_04 : Anti-duplication des témoignages. Chaque témoignage client génère une empreinte numérique unique (fingerprint). Le système refuse l'insertion d'un témoignage si son empreinte existe déjà en base.

RG_POR_05 : Unicité des éléments de présentation. Les sections "Photo de profil", "À propos de moi" et "Spécialité" sont des singletons : le système n'autorise qu'une seule instance active de ces éléments à la fois.

5. DOMAINE : MESSAGERIE EXTERNE (app_contact)
RG_MSG_01 : Validité des messages. Un visiteur externe peut envoyer un message via le formulaire de contact uniquement s'il renseigne un nom valide, une adresse email syntaxiquement correcte et un corps de message non vide.

RG_MSG_02 : Horodatage des contacts. Tout message reçu est automatiquement estampillé avec la date et l'heure système exactes de sa réception.

INTRODUCTION GÉNÉRALE
1. Contexte du Projet
Pour un développeur web, la possession d'une identité numérique forte est indispensable. C'est dans cette optique qu'est né ce projet : la création d'un Portfolio Dynamique couplé à un Tableau de Bord d'Administration centralisé. Plus qu'une simple vitrine statique, cette plateforme a été pensée comme un véritable Système d'Information (SI) capable de gérer dynamiquement du contenu, de sécuriser des accès multiples et d'analyser le trafic en temps réel.

2. Objectifs de la Plateforme
Le projet a été scindé en deux grands espaces fonctionnels répondant à des objectifs distincts :

Le Front-Office (L'espace public) : Une interface moderne et réactive servant de vitrine professionnelle. Elle permet d'exposer des compétences, de lister des réalisations (avec un système de "Démo Live"), d'afficher des témoignages clients et de recueillir des demandes via un formulaire de contact sécurisé.

Le Back-Office (L'espace d'administration) : Un tableau de bord privé et hautement sécurisé. Il intègre un moteur de contrôle d'accès basé sur les rôles (RBAC - Role Based Access Control), permettant de gérer finement les permissions (Matrice Rôles/Menus). De plus, il embarque un module de traçabilité complet (Audit) qui enregistre automatiquement les actions des utilisateurs et des visiteurs, tout en respectant les normes de conservation des données avec un système de purge automatisé.

3. Choix Technologiques
Pour répondre aux exigences de robustesse, de rapidité de développement et de sécurité, des technologies modernes et éprouvées ont été sélectionnées :

Le Framework Django (Python) : Choisi pour son architecture MVT (Model-View-Template) claire, son ORM (Object-Relational Mapping) puissant et ses mécanismes de sécurité intégrés (protection CSRF, hachage PBKDF2 pour les mots de passe).

Base de données SQLite : Utilisée pour sa légèreté et sa portabilité, elle s'intègre parfaitement aux environnements Django tout en supportant les contraintes relationnelles complexes définies dans notre modélisation.

Interface (UI/UX) : L'intégration de Bootstrap 5, associé à du JavaScript vanilla et jQuery, permet de garantir un affichage Responsive Design (adaptatif sur mobile et PC) et d'offrir une expérience utilisateur fluide (fenêtres modales, requêtes asynchrones, auto-fermeture des alertes).

4. Démarche Méthodologique : L'Analyse MERISE
Afin de garantir la fiabilité du code et l'évolutivité de la base de données, l'architecture de cette plateforme n'a pas été codée de manière empirique. Elle est le fruit d'une conception rigoureuse basée sur la méthode MERISE.

Ce présent dossier d'architecture technique retrace l'intégralité de ce processus de conception conceptuelle, organisationnelle, logique et physique. Il démontre comment les besoins métiers (Les Règles de Gestion) ont été traduits de manière scientifique en bases de données relationnelles optimisées (MCD, MLD, MPD) et en algorithmes de traitement sécurisés (MCT, MOT, MLT).
